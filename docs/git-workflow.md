# Регламент работы с ветками и коммитами

## Модель ветвления

Используем **GitHub Flow** (упрощённый git flow):

- `main` — защищённая ветка, всегда в рабочем состоянии
- Вся работа ведётся через feature-ветки
- Любое изменение в `main` — только через Pull Request
- Минимум 1 approve от другого участника команды
- CI должен быть зелёным перед merge

## Именование веток

Формат: `тип/номер-тикета-краткое-описание`

Примеры:
```
feat/12-user-auth
fix/34-email-validation
docs/5-update-glossary
refactor/18-prisma-queries
test/22-api-integration
chore/7-ci-setup
```

## Conventional Commits

Формат коммитов:

```
тип(область): краткое описание

Подробное описание (опционально)

Refs #номер-тикета
```

### Типы коммитов

| Тип | Описание |
|-----|----------|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `docs` | Изменения в документации |
| `refactor` | Рефакторинг без изменения поведения |
| `test` | Добавление/изменение тестов |
| `chore` | Настройка инфраструктуры, CI, зависимости |
| `style` | Форматирование, без изменения логики |

### Примеры коммитов

```
feat(backend): добавить эндпоинт поиска компаний

Реализован GET /api/companies/search с фильтрацией
по нише и городу.

Refs #12
```

```
fix(frontend): исправить валидацию email в форме контакта

Refs #34
```

## Pull Request

- Название PR должно отражать суть изменений
- В описании PR указать ссылку на тикет
- PR должен содержать осмысленные, атомарные изменения
- Не смешивать разные задачи в одном PR

## Защита ветки main

Настройки Branch Protection Rules:
- Require pull request reviews (min 1)
- Require status checks to pass before merging
- No direct push to main
