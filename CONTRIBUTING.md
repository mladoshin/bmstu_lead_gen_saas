# Как работать в проекте

## Начало работы

1. Склонируй репозиторий и установи зависимости (см. [README.md](README.md))
2. Ознакомься с [регламентом веток и коммитов](docs/git-workflow.md)
3. Убедись, что линтер и тесты проходят локально перед пушем

## Рабочий процесс

1. Выбери тикет из [GitHub Issues](../../issues) и назначь на себя
2. Создай feature-ветку от `main` (см. [именование веток](docs/git-workflow.md))
3. Напиши код и тесты
4. Убедись, что CI зелёный
5. Создай Pull Request и запроси ревью
6. После approve — merge в `main`

## Definition of Done

Задача считается выполненной, если:

- Код написан и покрыт тестами (если применимо)
- PR создан и прошёл code review (минимум 1 approve)
- CI успешно прошёл
- Документация обновлена (если менялись требования/архитектура)
- Тикет закрыт и связан с PR/коммитом
- Изменение попало в `main`

## Требования к тикетам

Каждый тикет должен содержать:

1. Краткое название (что делаем)
2. Описание (что должно получиться)
3. Acceptance Criteria (как понять, что готово)
4. Приоритет
5. Оценку в часах
6. Ссылку на ТЗ или раздел документации (если применимо)

## Требования к тестам

Минимум:
- Unit-тесты на ключевую бизнес-логику
- Хотя бы 1 интеграционный тест (API/БД)

Желательно:
- e2e тесты для основного сценария
- Тесты запускаются автоматически в CI

## Стиль кода

- Используем ESLint + Prettier
- Язык документации и комментариев — русский
- Названия переменных, функций, классов — на английском
